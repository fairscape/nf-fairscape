# nf-fairscape code walkthrough

A slow tour of how this plugin works, for someone who knows Python but not Groovy,
Nextflow, or its plugin system. Every section names the file it is talking about —
open them side by side with this document.

---

## Part 0 — A five-minute Groovy primer

Groovy is "Java with Python ergonomics". It compiles to the JVM, but you will
recognize almost everything. The idioms this codebase actually uses:

| Groovy | Python equivalent | Meaning |
| ------ | ----------------- | ------- |
| `a ?: b` | `a if a else b` | "Elvis operator" — fallback default |
| `a?.b` | `a.b if a is not None else None` | null-safe access |
| `[k: v]` | `{'k': v}` | map (dict) literal |
| `['a', 'b']` | `['a', 'b']` | list literal |
| `list.collect { x -> f(x) }` | `[f(x) for x in list]` | map |
| `list.findAll { x -> cond }` | `[x for x in list if cond]` | filter |
| `list.each { x -> ... }` | `for x in list:` | loop |
| `map.collectEntries { k, v -> [(k2): v2] }` | dict comprehension | rebuild a dict |
| `final x = ...` | `x = ...` | immutable local, type inferred |
| `"text ${expr}"` | f-string | string interpolation |
| `String name` / `Path path` | type annotation | Java-style declared types |

Annotations you'll see on classes:

- `@CompileStatic` — "type-check this like Java" (performance + safety; the default
  Groovy mode is dynamic like Python).
- `@Slf4j` — injects a `log` object (`log.warn`, `log.debug`).
- `@Delegate` — "forward unknown method calls to this field" (used so the renderer
  can call `normalizePath(...)` directly instead of `normalizer.normalizePath(...)`).

One trap that actually bit us during development: Groovy's `list.unique()` **mutates
in place** and throws on read-only collections; `unique(false)` returns a new list.
See `FairscapeRenderer.groovy` — every `.unique(false)` is there on purpose.

## Part 1 — What a Nextflow plugin even is

When you run `nextflow run . -plugins nf-fairscape@0.1.0`:

1. Nextflow looks in `~/.nextflow/plugins/` for a folder `nf-fairscape-0.1.0`
   (that's what `make install` created — an unzipped JAR of our compiled classes).
2. Inside is `META-INF/MANIFEST.MF` + `META-INF/spec.json`, generated by the build,
   which say: "the plugin class is `nextflow.prov.ProvPlugin`, and it exposes two
   *extension points*: `FairscapeConfig` and `ProvObserverFactory`."
3. Nextflow loads those classes and calls them at well-defined moments.

An **extension point** is just "a class Nextflow is allowed to discover and call".
Everything else in the plugin is private plumbing.

Where this is declared: **`build.gradle`** (repo root):

```groovy
nextflowPlugin {
    nextflowVersion = '25.10.0'                    // minimum Nextflow version
    className = 'nextflow.prov.ProvPlugin'         // the plugin's front door
    extensionPoints = [
        'nextflow.prov.FairscapeConfig',           // registers the config block
        'nextflow.prov.ProvObserverFactory'        // registers the event listener
    ]
}
```

> **Why is the package still called `nextflow.prov`?** This is a fork of nf-prov.
> Keeping the internal Java package name identical makes diffing/rebasing against
> upstream trivial, and package names are invisible to users. Only the plugin *id*
> (`nf-fairscape`) and config scope (`fairscape`) were rebranded.

### The source files, all of them

```
src/main/groovy/nextflow/prov/
├── ProvPlugin.groovy            (~10 lines)  front door, no logic
├── FairscapeConfig.groovy       (~120 lines) the `fairscape { }` config block
├── ProvObserverFactory.groovy   (~50 lines)  "do you want to observe this run?"
├── ProvObserver.groovy          (~130 lines) event listener, collects facts
├── Renderer.groovy              (~10 lines)  one-method interface
├── util/ProvHelper.groovy       (~150 lines) graph-derivation helpers (from nf-prov)
└── renderers/
    └── FairscapeRenderer.groovy (~350 lines) builds and writes the crate JSON
```

Call order at runtime:

```
nextflow startup
  └─> ProvObserverFactory.create(session)          reads config, or opts out
        └─> new ProvObserver(new FairscapeConfig)  hooks into the run
              ├─ onTaskComplete(event)   ...many times during the run
              ├─ onFilePublish(event)    ...many times during the run
              └─ onFlowComplete()        once, at the end
                    └─> FairscapeRenderer.render(...)   writes ro-crate-metadata.json
```

## Part 2 — The config block

**File: `src/main/groovy/nextflow/prov/FairscapeConfig.groovy`**

Nextflow validates config files strictly: any block it doesn't recognize is a hard
error ("Unknown config attribute"). The annotation at the top of this class is what
teaches Nextflow that `fairscape { ... }` is legal:

```groovy
@ScopeName('fairscape')                 // <- claims the `fairscape { }` scope
class FairscapeConfig implements ConfigScope {

    @ConfigOption                       // <- each option must be marked like this
    final String file
    ...
```

The constructor at the bottom of the file is where **every default** lives. It's
just Python's `opts.get(key, default)` in Groovy clothes:

```groovy
FairscapeConfig(Map opts) {
    file = opts.file ?: 'ro-crate-metadata.json'
    naan = opts.naan ?: '59853'
    keywords = opts.keywords as List<String> ?: ['nextflow', 'workflow']
    ...
}
```

This is why a pipeline with *no* `fairscape` block at all still produces a valid
crate: nothing in the required-fields chain can end up null. (Author falls back to
the workflow manifest, then the local username; license to Apache-2.0; etc. —
those manifest fallbacks are applied later, in the renderer, because the manifest
isn't available until the run exists.)

> **Gotcha (documented in README):** an *empty* `fairscape { }` block is rejected
> by Nextflow itself — an empty block is indistinguishable from a typo'd attribute
> read. Omit the block or set at least one option.

## Part 3 — The factory

**File: `src/main/groovy/nextflow/prov/ProvObserverFactory.groovy`**

At startup Nextflow calls `create(session)` on every registered observer factory.
Ours is ~10 real lines:

```groovy
Collection<TraceObserverV2> create(Session session) {
    final opts = session.config.fairscape as Map ?: Collections.emptyMap()
    final config = new FairscapeConfig(opts)
    if( !config.enabled )
        return null                      // plugin sits out this run
    return new ProvObserver(config)      // plugin is now wired into the run
}
```

`session` is the god-object of a Nextflow run. You'll see it everywhere. The parts
we use: `session.config` (your parsed nextflow.config as nested maps),
`session.params` (the `params { }` values), `session.uniqueId` (the run/session
UUID), and `session.workflowMetadata` (start/end times, script path, manifest,
command line, Nextflow version).

## Part 4 — The event listener

**File: `src/main/groovy/nextflow/prov/ProvObserver.groovy`**

This class implements Nextflow's `TraceObserverV2` interface — a bag of optional
callbacks the engine invokes as the run progresses. (The interface itself lives in
Nextflow's own source: `nextflow/trace/TraceObserverV2.groovy` in the Nextflow
repo — there's a checkout at `../nextflow` if you want to see all available hooks.)

The four callbacks we implement, and what each stores:

| Callback | Fired when | What we keep |
| -------- | ---------- | ------------ |
| `onFlowCreate(session)` | run starts | the `session`, for later |
| `onTaskComplete(event)` / `onTaskCached(event)` | a task finishes / is reused from `-resume` cache | the `TaskRun` object, into `Set<TaskRun> tasks` |
| `onFilePublish(event)` | a file is copied to your output dir | `publishedFiles[workDirSource] = publishedTarget` |
| `onFlowComplete()` | run ends | nothing — this is where rendering happens |

A `TaskRun` (Nextflow's class, see `nextflow/processor/TaskRun.groovy` in the
Nextflow checkout) is the complete record of one task execution: name, the resolved
script text, container image, input file list, output file list, and a unique
**task hash** (the same hash you see in the console as `[40/381f9b]`).

The end-of-run logic:

```groovy
void onFlowComplete() {
    if( !session.isSuccess() )
        return                            // failed run -> no crate
    for( final renderer : renderers ) {
        try {
            renderer.render(session, tasks, workflowOutputs, publishedFiles)
        }
        catch( Exception e ) { log.warn "..." }   // a broken crate never kills a run
    }
}
```

> **One deliberate deviation from upstream nf-prov**, in `onTaskComplete`: we skip
> tasks with `task.failed || task.aborted` instead of upstream's `!task.isSuccess()`.
> Reason: *native* tasks (`exec:` blocks — Groovy code instead of a bash script)
> never get an exit code, so `isSuccess()` is false for them and upstream silently
> drops them on fresh runs — but *includes* them on `-resume` (the cached path has
> no filter). That made fresh and resumed crates disagree. The flag check treats
> both paths the same. Comment is in the code at that spot.

**File: `src/main/groovy/nextflow/prov/Renderer.groovy`** — the interface between
observer and renderer. One method; this is the whole contract:

```groovy
void render(Session session, Set<TaskRun> tasks,
            Map<String,Path> workflowOutputs, Map<Path,Path> publishedFiles)
```

## Part 5 — Deriving the provenance graph

**File: `src/main/groovy/nextflow/prov/util/ProvHelper.groovy`** (inherited
unchanged from nf-prov)

Nothing tracks "task A's output fed task B" during the run. It's *derived* at the
end from file paths, by these three static helpers:

- `getTaskInputs(task)` → map of *stage name → source path* for a task's input files.
- `getTaskOutputs(task)` → list of output file paths.
- `getTaskLookup(tasks)` → **the key data structure**: a map of
  *output file path → the TaskRun that produced it*, built by walking every task's
  outputs.
- `getWorkflowInputs(tasks, lookup)` → every task input that appears in **no**
  task's outputs — i.e. files that came from outside the run (your input data,
  downloaded URLs).

So: if `REVERSE` read `list.txt` and no task produced `list.txt`, it's a workflow
input. If a second task had read `reversed.txt`, the lookup would say "REVERSE made
that", and both tasks would end up pointing at the *same* Dataset node. **Shared
file → shared identifier → that's the edge in the graph.** This is also exactly why
the WRROC "workflow plan" layer (steps, parameter connections) could be dropped in
the EVI mapping — the dataflow re-emerges from the files alone.

Also here: `getEncodingFormat(path)` (MIME type guessing) and
`checkFileOverwrite(path, overwrite)` (refuses to clobber an existing crate unless
`overwrite = true`).

## Part 6 — The renderer, section by section

**File: `src/main/groovy/nextflow/prov/renderers/FairscapeRenderer.groovy`**

This is the only file that knows what FAIRSCAPE/EVI looks like. `render()` runs
top to bottom in seven steps; the section comments in the file match this list.

**Step 0 — constants (top of file).** `EVI_CONTEXT` is a verbatim copy of
`DEFAULT_CONTEXT` from `fairscape_models/fairscape_models/fairscape_base.py` —
schema.org as the base vocabulary, the EVI and PROV namespaces, and the EVI edge
terms marked as `@id`-typed. The four `EVI_*` type constants are the ontology URIs.
Nothing is invented; every URI resolves.

**Step 1 — derive the graph.** First lines of `render()`: build `taskLookup` and
`workflowInputs` via ProvHelper (Part 5).

**Step 2 — resolve crate-level fields.** The fallback chains, e.g.:

```groovy
final author = (config.author ?: manifest.author ?: System.getProperty('user.name') ?: 'Unknown') as String
```

config wins → workflow `manifest { }` block → generated value. `ensureDescription()`
(bottom of file) guarantees the 10-character minimum that the pydantic models
enforce on descriptions.

**Step 3 — mint identifiers.** The heart of reproducibility. `mintArk()` (bottom
of the file, a static method so the unit test can hit it directly):

```
ark:{naan}/{prefix}-{slug(name)}-{sha1(stableSourceId)[0..6]}
```

What gets hashed is chosen so IDs are *deterministic*:

| Entity | hashed source |
| ------ | ------------- |
| root crate / run Computation | session id |
| task Computation | the task hash (stable across `-resume`) |
| workflow Software | normalized script path + version |
| Dataset (any file) | normalized file path |

`fileArk(path)` wraps minting for files with a cache map (`fileArks`) so one
physical file always gets exactly one ID — including the case where a published
file is the *same* file as a work-dir output (the publish map registers the target
path as an alias of the source's ARK).

> **Gotcha handled here:** a published file can have a **null source** — Nextflow
> itself writes some files (workflow-output index files) that no task produced.
> The registration loop mints from the target in that case. This was a real
> crash during development.

**Step 4 — build the nodes.** Each entity is a plain Groovy map (`[key: value]`),
assembled in this order — software, task Computations, the run Computation,
Datasets, then root + descriptor:

- **Task Computation** — one per collected `TaskRun`: `command` = the task's
  script, `usedSoftware` = its own process's Software node, `usedDataset` = its
  input files' ARKs, `generated` = its outputs' ARKs, `isPartOf` → the run
  Computation. Extra keys (`containerImage`, `identifier` = task hash) are not
  part of EVI but pydantic's `extra='allow'` accepts them.
- **Process Software** — one per process (e.g. `REVERSE`), `isPartOf` → the
  workflow-script Software, `contentUrl` = the file defining the process (found
  via Nextflow's `ScriptMeta`, so it points at the module file in modular
  pipelines). Only the *run* Computation references the whole script + engine.
- **Run Computation** — the whole run: launch `command`, params folded to
  `"name: value"` strings (`foldParams()`), external inputs, all published outputs,
  `startTime`/`endTime` as extras.
- **Dataset** — one per unique file. `generatedBy` points back at the producing
  Computation (the **inverse** of that Computation's `generated` — both directions
  are written, matching the LakeDB fixture in fairscape_models). `contentUrl` is
  crate-relative for published files, a normalized absolute path or original URL
  otherwise (remote inputs keep their https URL — `PathNormalizer`, a Nextflow
  utility class, does that translation).
- **Root** — `@type ["Dataset", EVI#ROCrate]`, `conformsTo` the FAIRSCAPE profile,
  `hasPart` listing *every* node above.

**Step 5 — write.** The last line of `render()`:

```groovy
path.text = JsonOutput.prettyPrint(JsonOutput.toJson(crate))
```

`.text =` on a `Path` is Groovy for "write this string to the file".

The node-by-node emission table (what fields each node carries and where each
value comes from) is in **`docs/FAIRSCAPE.md`** — that doc is the spec, this one
is the tour.

## Part 7 — Build, install, test

- **`build.gradle`** — plugin metadata (Part 1) plus the version read from
  **`VERSION`**. **`settings.gradle`** — project name, plus the *foojay toolchain
  resolver*, which auto-downloads the Java 21 that the Nextflow plugin framework
  requires (your machine only has Java 17; without this line the build fails with
  "Cannot find a Java installation... matching languageVersion=21").
- **`Makefile`** — the entry points you'll actually type:
  - `make assemble` / `make test` — compile / run unit tests
  - `make install` — build **and copy into `~/.nextflow/plugins/`** (required
    after *every* code change, or Nextflow keeps running the old version)
  - `make verify` — the full loop: install → run `nf-fairscape-test/` → validate
    the crate with fairscape_models
- **Unit tests** — `src/test/groovy/nextflow/prov/`, written in Spock (Groovy's
  test framework; `expect:`/`where:` blocks are parameterized tests, very readable).
  They cover ARK minting/slugging/description rules and the factory wiring.
- **The real acceptance test** — `nf-fairscape-test/validate_crate.py`: loads the
  emitted JSON into `fairscape_models.rocrate.ROCrateV1_2.model_validate` (your
  pydantic schema is the oracle) and checks that every `ark:` reference in the
  graph resolves to a node in the graph.
- **Examples** — `nf-fairscape-test/` (multi-process pipeline, remote URL inputs)
  and `examples/reverse-list/` (the minimal demo).

## Part 8 — How to make common changes

- **Add a config option:** add a `@ConfigOption final ...` field + a line in the
  constructor of `FairscapeConfig.groovy`, then use `config.<name>` in the renderer.
- **Add a field to every Computation:** find the task-computation `collect` block
  in `FairscapeRenderer.render()` and add a map entry. If it's not an EVI-defined
  field it still validates (extras allowed) — keep to schema.org names.
- **Change what counts as an ID:** `mintArk` / `fileArk` at the bottom of the
  renderer. Remember the unit tests pin the current format.
- After any change: `make install`, rerun a pipeline, and run `validate_crate.py`
  on the output. If a run says "Error occurred while rendering" the full stack
  trace is in `.nextflow.log` (the observer logs it at debug level).
